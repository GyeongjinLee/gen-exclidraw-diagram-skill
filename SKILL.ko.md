# Excalidraw 다이어그램 생성기

단순히 정보를 **표시**하는 것이 아니라 **시각적으로 논증하는** `.excalidraw` JSON 파일을 생성합니다.

**설정:** 사용자가 이 스킬(렌더러, 의존성 등)을 설정하도록 요청한 경우, 지침은 `README.md`를 참고하세요.

## 커스터마이징

**모든 색상과 브랜드별 스타일은 하나의 파일에 집중되어 있습니다:** `references/color-palette.md`. 다이어그램을 생성하기 전에 이 파일을 읽고, 도형 채우기, 선 색상, 텍스트 색상, 증거 아티팩트 배경 등 모든 색상 선택의 단일 소스로 활용하세요.

자신만의 브랜드 스타일로 다이어그램을 생성하려면 `color-palette.md`를 편집하세요. 이 파일의 나머지 부분은 모두 범용 디자인 방법론과 Excalidraw 모범 사례입니다.

## 핵심 철학

**다이어그램은 표시가 아니라 논증해야 합니다.**

다이어그램은 형식화된 텍스트가 아닙니다. 단어만으로는 표현할 수 없는 관계, 인과관계, 흐름을 보여주는 시각적 논증입니다. 도형이 곧 의미가 되어야 합니다.

**동형성 테스트**: 모든 텍스트를 제거했을 때, 구조만으로도 개념을 전달할 수 있습니까? 그렇지 않다면 다시 설계하세요.

**교육 테스트**: 이 다이어그램에서 누군가가 구체적인 무언가를 배울 수 있습니까, 아니면 단지 박스에 레이블을 붙인 것에 불과합니까? 좋은 다이어그램은 가르칩니다 — 실제 형식, 실제 이벤트 이름, 구체적인 예시를 보여줍니다.

## 깊이 평가 (먼저 수행)

설계하기 전에 이 다이어그램에 필요한 세부 수준을 결정합니다:

### 단순/개념적 다이어그램
다음의 경우 추상적인 도형을 사용합니다:
- 멘탈 모델이나 철학을 설명할 때
- 대상이 기술적 세부 사항을 필요로 하지 않을 때
- 개념 자체가 추상화일 때 (예: "관심사의 분리")

### 포괄적/기술적 다이어그램
다음의 경우 구체적인 예시를 사용합니다:
- 실제 시스템, 프로토콜, 또는 아키텍처를 다이어그램으로 그릴 때
- 교육하거나 설명하기 위해 다이어그램을 사용할 때 (예: YouTube 영상)
- 대상이 실제로 어떻게 생겼는지 이해해야 할 때
- 여러 기술이 어떻게 통합되는지 보여줄 때

**기술 다이어그램의 경우, 반드시 증거 아티팩트를 포함해야 합니다** (아래 참조).

## 연구 의무 (기술 다이어그램용)

**기술적인 내용을 그리기 전에 실제 사양을 연구하세요.**

프로토콜, API, 또는 프레임워크를 다이어그램으로 그릴 경우:
1. 실제 JSON/데이터 형식을 조회합니다
2. 실제 이벤트 이름, 메서드 이름, 또는 API 엔드포인트를 찾습니다
3. 각 요소들이 실제로 어떻게 연결되는지 이해합니다
4. 일반적인 플레이스홀더가 아닌 실제 용어를 사용합니다

나쁜 예: "Protocol" → "Frontend"
좋은 예: "AG-UI streams events (RUN_STARTED, STATE_DELTA, A2UI_UPDATE)" → "CopilotKit renders via createA2UIMessageRenderer()"

**연구를 통해 다이어그램이 정확하고 교육적으로 만들어집니다.**

## 증거 아티팩트

증거 아티팩트는 다이어그램의 정확성을 증명하고 시청자가 배울 수 있도록 돕는 구체적인 예시입니다. 기술 다이어그램에 포함하세요.

**증거 아티팩트 유형** (다이어그램에 관련된 것을 선택):

| 아티팩트 유형 | 사용 시기 | 렌더링 방법 |
|--------------|---------|------------|
| **코드 스니펫** | API, 통합, 구현 세부 사항 | 어두운 사각형 + 구문 색상 텍스트 (색상 팔레트에서 증거 아티팩트 색상 참조) |
| **데이터/JSON 예시** | 데이터 형식, 스키마, 페이로드 | 어두운 사각형 + 색상 텍스트 (색상 팔레트 참조) |
| **이벤트/단계 순서** | 프로토콜, 워크플로우, 라이프사이클 | 타임라인 패턴 (선 + 점 + 레이블) |
| **UI 목업** | 실제 출력/결과 보여주기 | 실제 UI를 모방한 중첩 사각형 |
| **실제 입력 내용** | 시스템에 들어가는 것 보여주기 | 샘플 내용이 보이는 사각형 |
| **API/메서드 이름** | 실제 함수 호출, 엔드포인트 | 플레이스홀더가 아닌 문서의 실제 이름 사용 |

**예시**: 스트리밍 프로토콜에 대한 다이어그램의 경우:
- 사양의 실제 이벤트 이름 표시 ("이벤트 1", "이벤트 2"가 아닌)
- 연결 방법을 보여주는 코드 스니펫
- 스트리밍된 데이터가 실제로 어떻게 생겼는지

**예시**: 데이터 변환 파이프라인에 대한 다이어그램의 경우:
- 샘플 입력 데이터 표시 (실제 형식, "Input"이 아닌)
- 샘플 출력 데이터 표시 (실제 형식, "Output"이 아닌)
- 관련이 있는 경우 중간 상태 표시

핵심 원칙: 이름만 표시하는 것이 아니라 **실제로 어떻게 생겼는지 보여주세요**.

## 다중 줌 아키텍처

포괄적인 다이어그램은 여러 줌 레벨에서 동시에 작동합니다. 국경선과 거리 이름을 모두 보여주는 지도처럼 생각하세요.

### 레벨 1: 요약 흐름
전체 파이프라인이나 프로세스를 한눈에 보여주는 간단한 개요. 종종 다이어그램의 상단이나 하단에 배치됩니다.

*예시*: `입력 → 처리 → 출력` 또는 `클라이언트 → 서버 → 데이터베이스`

### 레벨 2: 섹션 경계
관련 구성 요소를 그룹화하는 레이블이 있는 영역. 시청자가 무엇이 함께 속하는지 이해하는 데 도움이 되는 시각적 "방"을 만듭니다.

*예시*: 책임별 그룹화 (백엔드/프론트엔드), 단계별 (설정/실행/정리), 또는 팀별 (사용자/시스템/외부)

### 레벨 3: 섹션 내부 세부 사항
각 섹션 내의 증거 아티팩트, 코드 스니펫, 구체적인 예시. 여기에 교육적 가치가 있습니다.

*예시*: "백엔드" 섹션 내에서 "API 응답"이라는 레이블의 박스가 아닌 실제 API 응답 형식을 보여줄 수 있습니다.

**포괄적인 다이어그램의 경우 세 가지 레벨을 모두 포함하세요.** 요약은 맥락을 제공하고, 섹션은 구성하고, 세부 사항은 가르칩니다.

### 나쁜 예 vs 좋은 예

| 나쁜 예 (표시) | 좋은 예 (논증) |
|----------------|----------------|
| 레이블이 있는 5개의 동일한 박스 | 각 개념이 행동을 반영하는 도형을 가짐 |
| 카드 그리드 레이아웃 | 시각적 구조가 개념적 구조와 일치 |
| 텍스트를 장식하는 아이콘 | 의미 자체인 도형 |
| 모든 것에 동일한 컨테이너 | 개념별 뚜렷한 시각적 어휘 |
| 모든 것이 박스 안에 | 선택적 컨테이너와 함께 자유 부동 텍스트 |

### 단순 vs 포괄적 (어느 것이 필요한지 파악)

| 단순 다이어그램 | 포괄적 다이어그램 |
|----------------|-----------------|
| 일반 레이블: "입력" → "처리" → "출력" | 구체적: 입력/출력이 실제로 어떻게 생겼는지 보여줌 |
| 이름이 있는 박스: "API", "Database", "Client" | 이름이 있는 박스 + 실제 요청/응답 예시 |
| "이벤트" 또는 "메시지" 레이블 | 사양의 실제 이벤트/메시지 이름이 있는 타임라인 |
| "UI" 또는 "Dashboard" 사각형 | 실제 UI 요소와 내용을 보여주는 목업 |
| 설명에 ~30초 | ~2-3분의 교육 내용 |
| 시청자가 구조를 배움 | 시청자가 구조와 세부 사항을 모두 배움 |

**단순 다이어그램**은 추상적인 개념, 빠른 개요, 또는 대상이 이미 세부 사항을 알고 있는 경우에 적합합니다. **포괄적인 다이어그램**은 기술 아키텍처, 튜토리얼, 교육 내용, 또는 다이어그램 자체가 가르치기를 원할 때 필요합니다.

## 컨테이너 vs. 자유 부동 텍스트

**모든 텍스트에 도형이 필요한 것은 아닙니다.** 자유 부동 텍스트를 기본으로 사용하세요. 컨테이너는 목적을 제공할 때만 추가합니다.

| 컨테이너를 사용할 때 | 자유 부동 텍스트를 사용할 때 |
|--------------------|-----------------------------|
| 섹션의 초점 요소일 때 | 레이블이나 설명일 때 |
| 다른 요소와 시각적 그룹화가 필요할 때 | 보조 세부 사항이나 메타데이터일 때 |
| 화살표가 연결되어야 할 때 | 주변의 무언가를 설명할 때 |
| 도형 자체가 의미를 담고 있을 때 (결정 다이아몬드 등) | 타이포그래피만으로 충분한 계층 구조를 만들 때 |
| 시스템에서 구별되는 "사물"을 나타낼 때 | 섹션 제목, 부제목, 또는 주석일 때 |

**계층 구조로서의 타이포그래피**: 박스 없이 글꼴 크기, 굵기, 색상을 사용하여 시각적 계층 구조를 만드세요. 28px 제목은 사각형으로 둘러쌀 필요가 없습니다.

**컨테이너 테스트**: 각 박스 요소에 대해 "이것이 자유 부동 텍스트로 작동할까?"라고 물어보세요. 그렇다면 컨테이너를 제거하세요.

## 설계 프로세스 (JSON 생성 전에 수행)

### 단계 0: 필요한 깊이 평가
무엇보다 먼저, 이것이 필요한지 결정합니다:
- **단순/개념적**: 추상적인 도형, 레이블, 관계 (멘탈 모델, 철학)
- **포괄적/기술적**: 구체적인 예시, 코드 스니펫, 실제 데이터 (시스템, 아키텍처, 튜토리얼)

**포괄적인 경우**: 먼저 연구하세요. 실제 사양, 형식, 이벤트 이름, API를 조회하세요.

### 단계 1: 깊이 이해
내용을 읽으세요. 각 개념에 대해 다음을 물어보세요:
- 이 개념은 무엇을 **합니까**? (그것이 무엇인지가 아닌)
- 개념들 사이에 어떤 관계가 있습니까?
- 핵심 변환이나 흐름은 무엇입니까?
- **이것을 이해하기 위해 누군가가 무엇을 봐야 합니까?** (단지 읽는 것이 아닌)

### 단계 2: 개념을 패턴에 매핑
각 개념에 대해 행동을 반영하는 시각적 패턴을 찾으세요:

| 개념이... | 이 패턴 사용 |
|-----------|-------------|
| 여러 출력을 생성함 | **팬아웃** (중심에서 방사형 화살표) |
| 입력을 하나로 결합함 | **수렴** (깔때기, 합쳐지는 화살표) |
| 계층/중첩을 가짐 | **트리** (선 + 자유 부동 텍스트) |
| 단계의 순서임 | **타임라인** (선 + 점 + 자유 부동 레이블) |
| 지속적으로 루프하거나 개선됨 | **나선/사이클** (시작점으로 돌아오는 화살표) |
| 추상적인 상태나 맥락임 | **클라우드** (겹치는 타원) |
| 입력을 출력으로 변환함 | **조립 라인** (이전 → 처리 → 이후) |
| 두 가지를 비교함 | **나란히** (대조가 있는 병렬) |
| 단계로 분리됨 | **간격/분리** (섹션 사이의 시각적 분리) |

### 단계 3: 다양성 확보
다중 개념 다이어그램의 경우: **각 주요 개념은 다른 시각적 패턴을 사용해야 합니다**. 일정한 카드나 그리드는 안 됩니다.

### 단계 4: 흐름 스케치
JSON 전에 눈이 다이어그램을 어떻게 이동하는지 정신적으로 추적하세요. 명확한 시각적 스토리가 있어야 합니다.

### 단계 5: JSON 생성
이제 Excalidraw 요소를 만드세요. **대형 다이어그램 처리 방법은 아래를 참조하세요.**

### 단계 6: 렌더링 & 검증 (필수)
JSON을 생성한 후, 다이어그램이 올바르게 보일 때까지 렌더-보기-수정 루프를 반드시 실행해야 합니다. 이것은 선택 사항이 아닙니다 — 전체 과정은 아래 **렌더링 & 검증** 섹션을 참조하세요.

## 대형/포괄적 다이어그램 전략

**포괄적이거나 기술적인 다이어그램의 경우, JSON을 한 섹션씩 구축해야 합니다.** 단일 패스에서 전체 파일을 생성하려고 시도하지 마세요. 이것은 엄격한 제약입니다 — Claude Code는 응답당 ~32,000 토큰 출력 한도가 있으며, 포괄적인 다이어그램은 한 번에 쉽게 초과합니다. 그렇지 않더라도, 한꺼번에 생성하면 품질이 떨어집니다. 섹션별 방식이 모든 면에서 더 좋습니다.

### 섹션별 워크플로우

**단계 1: 각 섹션 구축**

1. **기본 파일 생성**: JSON 래퍼(`type`, `version`, `appState`, `files`)와 첫 번째 섹션의 요소를 포함합니다.
2. **편집당 하나의 섹션 추가.** 각 섹션은 전용 패스를 갖습니다 — 시간을 충분히 사용하세요. 레이아웃, 간격, 그리고 이 섹션이 기존 내용과 어떻게 연결되는지 신중하게 생각하세요.
3. **설명적인 문자열 ID 사용** (예: `"trigger_rect"`, `"arrow_fan_left"`) 하여 섹션 간 참조를 읽기 쉽게 합니다.
4. **섹션별 시드 네임스페이스 지정** (예: 섹션 1은 100xxx, 섹션 2는 200xxx 사용)하여 충돌을 피합니다.
5. **섹션 간 바인딩을 진행하면서 업데이트합니다.** 새 섹션의 요소가 이전 섹션의 요소에 바인딩되어야 할 때 (예: 섹션을 연결하는 화살표), 동시에 이전 요소의 `boundElements` 배열을 편집합니다.

**단계 2: 전체 검토**

모든 섹션이 준비된 후, 전체 JSON을 읽고 확인합니다:
- 섹션 간 화살표가 양쪽에 올바르게 바인딩되어 있습니까?
- 전체 간격이 균형 잡혀 있습니까, 아니면 일부 섹션은 좁고 다른 섹션은 여백이 너무 많습니까?
- ID와 바인딩이 실제로 존재하는 요소를 참조합니까?

렌더링 전에 정렬 또는 바인딩 문제를 수정하세요.

**단계 3: 렌더링 & 검증**

이제 렌더링 & 검증 섹션의 렌더-보기-수정 루프를 실행합니다. 여기에서 JSON에서는 명확하지 않은 시각적 문제 — 겹침, 클리핑, 불균형한 구성 — 을 발견합니다.

### 섹션 경계

다이어그램 계획의 자연스러운 시각적 그룹화를 중심으로 섹션을 계획하세요. 일반적인 대형 다이어그램은 다음과 같이 나눌 수 있습니다:

- **섹션 1**: 진입점 / 트리거
- **섹션 2**: 첫 번째 결정 또는 라우팅
- **섹션 3**: 주요 내용 (영웅 섹션 — 가장 큰 단일 섹션일 수 있음)
- **섹션 4-N**: 나머지 단계, 출력 등

각 섹션은 독립적으로 이해 가능해야 합니다: 해당 요소, 내부 화살표, 인접 섹션에 대한 모든 상호 참조.

### 하지 말아야 할 것

- **한 응답에서 전체 다이어그램을 생성하지 마세요.** 출력 토큰 한도에 도달하여 잘린 손상된 JSON을 생성합니다. 다이어그램이 충분히 작더라도, 섹션으로 나누면 더 나은 결과를 얻습니다.
- **JSON을 생성하기 위해 코딩 에이전트를 사용하지 마세요.** 에이전트는 스킬 규칙에 대한 충분한 맥락을 갖지 못하며, 조정 오버헤드가 어떤 이점도 무효화합니다.
- **Python 생성기 스크립트를 작성하지 마세요.** 템플릿화와 좌표 계산이 도움이 될 것 같지만 디버깅을 어렵게 만드는 간접적인 계층을 도입합니다. 설명적인 ID가 있는 수작업 JSON이 더 유지 보수하기 쉽습니다.

## 시각적 패턴 라이브러리

### 팬아웃 (일대다)
여러 대상으로 방사되는 화살표가 있는 중앙 요소. 사용 대상: 소스, PRD, 근본 원인, 중앙 허브.
```
        ○
       ↗
  □ → ○
       ↘
        ○
```

### 수렴 (다대일)
단일 출력으로 화살표를 통해 합쳐지는 여러 입력. 사용 대상: 집계, 깔때기, 합성.
```
  ○ ↘
  ○ → □
  ○ ↗
```

### 트리 (계층)
연결선과 자유 부동 텍스트가 있는 부모-자식 분기 (박스 불필요). 사용 대상: 파일 시스템, 조직도, 분류법.
```
  label
  ├── label
  │   ├── label
  │   └── label
  └── label
```
트렁크와 브랜치에는 `line` 요소를 사용하고, 레이블에는 자유 부동 텍스트를 사용합니다.

### 나선/사이클 (지속적인 루프)
시작점으로 돌아오는 화살표가 있는 순서의 요소. 사용 대상: 피드백 루프, 반복 프로세스, 진화.
```
  □ → □
  ↑     ↓
  □ ← □
```

### 클라우드 (추상적 상태)
다양한 크기의 겹치는 타원. 사용 대상: 맥락, 메모리, 대화, 정신 상태.

### 조립 라인 (변환)
명확한 이전/이후가 있는 입력 → 처리 박스 → 출력. 사용 대상: 변환, 처리, 변환.
```
  ○○○ → [PROCESS] → □□□
  chaos              order
```

### 나란히 (비교)
시각적 대조가 있는 두 개의 병렬 구조. 사용 대상: 이전/이후, 옵션, 트레이드오프.

### 간격/분리 (경계)
섹션 사이의 시각적 공백 또는 장벽. 사용 대상: 단계 변경, 맥락 재설정, 경계.

### 구조로서의 선
박스 대신 선(type: `line`, 화살표가 아님)을 기본 구조 요소로 사용합니다:
- **타임라인**: 수직 또는 수평 선에 간격으로 작은 점 (10-20px 타원), 각 점 옆에 자유 부동 레이블
- **트리 구조**: 수직 트렁크 선 + 수평 브랜치 선, 자유 부동 텍스트 레이블 (박스 불필요)
- **구분선**: 섹션을 분리하는 얇은 점선
- **흐름 척추**: 박스를 연결하는 대신 요소들이 관계를 맺는 중앙 선

```
타임라인:               트리:
  ●─── 레이블 1          │
  │                      ├── 항목
  ●─── 레이블 2          │   ├── 하위
  │                      │   └── 하위
  ●─── 레이블 3          └── 항목
```

선 + 자유 부동 텍스트는 종종 박스 + 포함된 텍스트보다 더 깔끔한 결과를 만듭니다.

## 도형 의미

표현하는 것에 따라 도형을 선택하거나 도형을 전혀 사용하지 마세요:

| 개념 유형 | 도형 | 이유 |
|----------|------|------|
| 레이블, 설명, 세부 사항 | **없음** (자유 부동 텍스트) | 타이포그래피가 계층 구조를 만듦 |
| 섹션 제목, 주석 | **없음** (자유 부동 텍스트) | 글꼴 크기/굵기로 충분 |
| 타임라인의 마커 | 작은 `ellipse` (10-20px) | 시각적 앵커, 컨테이너가 아님 |
| 시작, 트리거, 입력 | `ellipse` | 부드러운, 기원 같은 |
| 끝, 출력, 결과 | `ellipse` | 완성, 목적지 |
| 결정, 조건 | `diamond` | 클래식 결정 기호 |
| 프로세스, 행동, 단계 | `rectangle` | 포함된 행동 |
| 추상적 상태, 맥락 | 겹치는 `ellipse` | 흐릿한, 클라우드 같은 |
| 계층 노드 | 선 + 텍스트 (박스 없음) | 선을 통한 구조 |

**규칙**: 컨테이너 없음을 기본으로 사용합니다. 의미를 담고 있을 때만 도형을 추가합니다. 텍스트 요소의 30% 미만이 컨테이너 안에 있도록 목표로 합니다.

## 의미로서의 색상

색상은 장식이 아닌 정보를 인코딩합니다. 모든 색상 선택은 `references/color-palette.md`에서 가져와야 합니다 — 의미적 도형 색상, 텍스트 계층 색상, 증거 아티팩트 색상이 모두 정의되어 있습니다.

**핵심 원칙:**
- 각 의미적 목적 (시작, 끝, 결정, AI, 오류 등)에는 특정 채우기/선 쌍이 있습니다
- 자유 부동 텍스트는 계층 구조에 색상을 사용합니다 (제목, 부제목, 세부 사항 — 각각 다른 레벨)
- 증거 아티팩트 (코드 스니펫, JSON 예시)는 자체 어두운 배경 + 색상 텍스트 체계를 사용합니다
- 대비를 위해 항상 더 어두운 선과 더 밝은 채우기를 쌍으로 사용합니다

**새로운 색상을 만들지 마세요.** 개념이 기존 의미적 카테고리에 맞지 않으면 Primary/Neutral 또는 Secondary를 사용합니다.

## 현대적인 미학

깔끔하고 전문적인 다이어그램을 위해:

### 거칠기
- `roughness: 0` — 깔끔하고 선명한 가장자리. 현대/기술 다이어그램에 사용합니다.
- `roughness: 1` — 손으로 그린 것 같은 유기적인 느낌. 브레인스토밍/비공식 다이어그램에 사용합니다.

**대부분의 전문적인 사용 사례에는 0을 기본으로 사용합니다.**

### 선 굵기
- `strokeWidth: 1` — 얇고 우아함. 선, 구분선, 미묘한 연결에 좋음.
- `strokeWidth: 2` — 표준. 도형과 기본 화살표에 좋음.
- `strokeWidth: 3` — 굵음. 강조를 위해 드물게 사용 (주요 흐름 선, 핵심 연결).

### 불투명도
**모든 요소에 `opacity: 100`을 항상 사용합니다.** 투명도 대신 색상, 크기, 선 굵기로 계층 구조를 만드세요.

### 도형 대신 작은 마커
전체 도형 대신 작은 점 (10-20px 타원)을 사용합니다:
- 타임라인 마커
- 글머리 기호
- 연결 노드
- 자유 부동 텍스트를 위한 시각적 앵커

## 렌더링 & 검증 (필수)

JSON만으로는 다이어그램을 판단할 수 없습니다. Excalidraw JSON을 생성하거나 편집한 후, PNG로 렌더링하고, 이미지를 보고, 올바를 때까지 발견한 것을 수정해야 합니다 — 루프로. 이것은 최종 확인이 아니라 워크플로우의 핵심 부분입니다.

### 렌더링 방법

```bash
cd .claude/skills/excalidraw-diagram/references && uv run python render_excalidraw.py <path-to-file.excalidraw>
```

이것은 `.excalidraw` 파일 옆에 PNG를 출력합니다. 그런 다음 **Read 도구**를 사용하여 실제로 봅니다.

### 루프

초기 JSON을 생성한 후 이 사이클을 실행합니다:

**1. 렌더링 & 보기** — 렌더 스크립트를 실행한 다음 PNG를 Read합니다.

**2. 원래 비전에 대한 감사** — 버그를 찾기 전에, 렌더링된 결과를 설계한 것과 비교합니다. 다음을 물어보세요:
- 시각적 구조가 계획한 개념적 구조와 일치합니까?
- 각 섹션이 의도한 패턴을 사용합니까 (팬아웃, 수렴, 타임라인 등)?
- 눈이 설계한 순서대로 다이어그램을 통해 흘러갑니까?
- 시각적 계층 구조가 올바릅니까 — 영웅 요소가 지배적이고 지원 요소가 더 작습니까?
- 기술 다이어그램의 경우: 증거 아티팩트 (코드 스니펫, 데이터 예시)가 읽기 쉽고 올바르게 배치되어 있습니까?

**3. 시각적 결함 확인:**
- 컨테이너에 의해 잘리거나 넘치는 텍스트
- 다른 요소와 겹치는 텍스트 또는 도형
- 요소를 통해 라우팅하는 대신 통과하는 화살표
- 잘못된 요소에 연결되거나 빈 공간을 가리키는 화살표
- 설명하는 것에 명확하게 앵커되지 않고 애매하게 떠 있는 레이블
- 고르게 간격을 두어야 하는 요소 사이의 불균등한 간격
- 너무 좁은 섹션 옆에 여백이 너무 많은 섹션
- 렌더링된 크기에서 읽기 어려운 너무 작은 텍스트
- 전체 구성이 치우쳐 있거나 불균형하게 느껴짐

**4. 수정** — 발견한 모든 것을 해결하기 위해 JSON을 편집합니다. 일반적인 수정:
- 텍스트가 잘릴 때 컨테이너를 넓힘
- 간격 및 정렬을 수정하기 위해 `x`/`y` 좌표 조정
- 요소 주위로 라우팅하기 위해 화살표 `points` 배열에 중간 웨이포인트 추가
- 설명하는 요소에 더 가깝게 레이블 재배치
- 섹션 간 시각적 무게를 재조정하기 위해 요소 크기 조정

**5. 재렌더링 & 재보기** — 렌더 스크립트를 다시 실행하고 새 PNG를 Read합니다.

**6. 반복** — 다이어그램이 비전 확인 (단계 2)과 결함 확인 (단계 3)을 모두 통과할 때까지 계속 순환합니다. 일반적으로 2-4번의 반복이 필요합니다. 심각한 버그가 없다고 해서 한 번의 패스 후에 중단하지 마세요 — 구성이 더 좋아질 수 있다면 개선하세요.

5. **교육적 가치**: 누군가가 이것에서 구체적인 무언가를 배울 수 있습니까?

### 개념적
6. **동형성**: 각 시각적 구조가 개념의 행동을 반영합니까?
7. **논증**: 다이어그램이 텍스트만으로는 표현할 수 없는 것을 보여줍니까?

### 컨테이너 규율
10. **최소한의 컨테이너**: 박스 요소가 자유 부동 텍스트로 작동할 수 있습니까?
11. **구조로서의 선**: 트리/타임라인 패턴이 박스 대신 선 + 텍스트를 사용합니까?
12. **타이포그래피 계층 구조**: 글꼴 크기와 색상이 시각적 계층 구조를 만들고 있습니까 (박스의 필요성 감소)?

### 구조적
13. **연결**: 모든 관계에 화살표 또는 선이 있음
14. **흐름**: 눈이 따라갈 명확한 시각적 경로
15. **계층 구조**: 중요한 요소가 더 크거나 더 고립되어 있음

### 기술적
16. **텍스트 정리**: `text`에 읽을 수 있는 단어만 포함
17. **글꼴**: `fontFamily: 3`
18. **거칠기**: 깔끔/현대적 스타일에는 `roughness: 0` (손으로 그린 스타일이 요청되지 않는 한)
19. **불투명도**: 모든 요소에 `opacity: 100` (투명도 없음)
20. **컨테이너 비율**: 텍스트 요소의 30% 미만이 컨테이너 안에 있어야 함

### 시각적 검증 (렌더링 필수)
21. **PNG로 렌더링됨**: 다이어그램이 렌더링되고 시각적으로 검사됨
22. **텍스트 오버플로 없음**: 모든 텍스트가 컨테이너에 맞음
23. **겹치는 요소 없음**: 도형과 텍스트가 의도치 않게 겹치지 않음
24. **균등한 간격**: 유사한 요소가 일정한 간격을 가짐
25. **화살표가 올바르게 연결됨**: 화살표가 다른 요소를 가로지르지 않고 의도된 요소에 연결됨
26. **내보내기 크기에서 읽을 수 있음**: 텍스트가 렌더링된 PNG에서 읽을 수 있음
27. **균형 잡힌 구성**: 큰 빈 공간이나 과밀한 영역 없음
